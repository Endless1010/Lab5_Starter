**1) Why is it important to put thought into your IDs & Classes when it comes to technology intersections? (e.g. how HTML, CSS, and JS intersect)**

IDs and classes are the primary bridge between HTML, CSS, and JavaScript. If you choose vague or inconsistent names (e.g., `div1`, `temp`), your CSS selectors become fragile and your JavaScript queries become hard to read and maintain. A well-named ID like `#volume-controls` is self-documenting and makes it immediately clear what the element represents across all three layers. Renaming an ID later requires a coordinated update across HTML structure, CSS rules, and JS selectors — so poor naming choices compound into significant technical debt. Clear, semantic names also prevent accidental style or behavior collisions when multiple developers work on the same project.

**2) What are Data attributes? Why might they be useful? How do you access them? What are the implications of using Data attributes when it comes to things like microdata?**

Data attributes are custom HTML attributes prefixed with `data-` (e.g., `data-horn-type="air-horn"`). They let you embed arbitrary metadata directly in an HTML element without using non-standard attributes or hidden fields. They're useful when JavaScript needs extra context about an element beyond its class or ID — for example, storing an item's ID or configuration value in the markup itself. You access them in JavaScript via `element.dataset.hornType` (camelCase of the suffix) or `element.getAttribute('data-horn-type')`. The implication around microdata is that data attributes and microdata (e.g., Schema.org using `itemprop`) serve different purposes — data attributes are for private, developer-defined metadata meant for scripts, while microdata is for semantic meaning readable by search engines and other consumers. Mixing them or using data attributes where microdata is expected means that semantic consumers will ignore the information, reducing SEO and accessibility benefits.

**3) What is a DOM fragment? Why are they powerful?**

A DOM fragment (`DocumentFragment`) is a lightweight, in-memory container that can hold a tree of DOM nodes without being part of the live document. You build up a subtree inside the fragment, then append the whole fragment to the document in one operation. This is powerful because DOM mutations (adding/removing elements) trigger reflows and repaints, which are expensive. By batching all your insertions inside a fragment first, you cause only a single reflow when the fragment is attached, rather than one reflow per element added. This makes building large lists or complex structures much more performant.

**4) What is the point of a "Virtual DOM"? What do you gain? What do you lose?**

A Virtual DOM (used by frameworks like React) is an in-memory JavaScript representation of the real DOM. When state changes, the framework re-renders the new Virtual DOM, diffs it against the previous version, and applies only the minimal set of real DOM updates needed. You gain a declarative programming model — you describe what the UI should look like for a given state and let the framework figure out the mutations — which leads to more predictable and testable code. You lose some raw performance: the diffing algorithm and the extra abstraction layer add overhead compared to carefully hand-written direct DOM manipulation. For very simple or highly performance-critical UIs, the Virtual DOM overhead may not be worth it.

**5) In JavaScript, usually you can reference every attribute of an element with a dot selector followed by the attribute name, except for the class attribute, which is className. Why is this so?**

`class` is a reserved keyword in JavaScript (used for ES6 class declarations). When the DOM API was designed in the early days of the web, JavaScript did not allow reserved words as property names on objects, so the property was named `className` instead to avoid a syntax error. Modern JavaScript technically allows reserved words as property names in object literals, but the DOM API kept `className` for backward compatibility. (The newer `classList` API was introduced as a more ergonomic alternative.)

**6) What is the difference between using addEventListener() and something like onClick() ? What are the advantages / disadvantages of both?**

`addEventListener('click', handler)` registers a handler without overwriting any previously registered handlers on the same event — you can attach multiple listeners for the same event type. It also gives you access to the options object (e.g., `{ once: true }`, `{ capture: true }`) for fine-grained control, and provides a clean way to remove the listener with `removeEventListener`. The inline `onclick` property (or the HTML attribute `onclick="..."`) is a single-slot assignment: assigning a new function replaces the old one, so you can only have one handler at a time. The HTML attribute form also mixes behavior into markup, making the code harder to maintain and test. In practice, `addEventListener` is preferred for all but the simplest cases because it is non-destructive, more powerful, and keeps JavaScript out of HTML.
